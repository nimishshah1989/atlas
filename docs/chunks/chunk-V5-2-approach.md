# Chunk V5-2 Approach: Alembic Migration + SQLAlchemy Models for V5 Agent Tables

## Actual data scale
- de_rs_scores: 14.7M rows (JIP read-only, not touched)
- atlas_intelligence: ~0-100 rows (new, no scale concern)
- New tables: atlas_agent_scores, atlas_agent_weights, atlas_agent_memory — start at 0 rows
- All new tables trivially small, no performance concerns

## HNSW index status
- `idx_intel_embedding_hnsw` already exists (m=16, ef_construction=64, vector_cosine_ops)
- `idx_intelligence_embedding` also already exists (duplicate HNSW)
- Migration uses `CREATE INDEX IF NOT EXISTS` — fully idempotent

## Chosen approach
- SQLAlchemy 2.0 `mapped_column()` syntax — matches existing pattern in models.py
- Alembic migration hand-written (not autogenerated) — avoids spurious index drops per `alembic-autogenerate-index-drift` bug pattern
- BIGSERIAL primary key for atlas_agent_scores and atlas_agent_memory (matches spec DDL, not UUID — these are high-write append tables)
- VARCHAR(100) primary key for atlas_agent_weights (agent_id is the natural key per spec)
- CheckConstraint on weight: `weight >= 0.3 AND weight <= 2.5` — punch list item 4 (rejects 0.2 and 3.0)

## Wiki patterns checked
- `alembic-mypy-attr-defined`: Use `# type: ignore[attr-defined]` on alembic op/context imports
- `alembic-autogenerate-index-drift`: Hand-write migration, no autogenerate
- `decimal-not-float`: All Numeric columns use Decimal mapped type

## Existing code being reused
- `backend/db/models.py`: Following exact same `mapped_column()` pattern as AtlasIntelligence, AtlasDecision
- `tests/db/test_intelligence_schema.py`: Following same pattern for DB schema tests with psycopg2

## Standard columns added (per database.md convention)
All atlas_* tables must have: id, created_at, updated_at, is_deleted, deleted_at
- atlas_agent_scores: spec has id (BIGSERIAL) + created_at. Adding updated_at, is_deleted, deleted_at.
- atlas_agent_weights: spec has updated_at. Adding created_at, is_deleted, deleted_at.
- atlas_agent_memory: spec has id (BIGSERIAL) + created_at. Adding updated_at, is_deleted, deleted_at.

## Edge cases
- CHECK constraint rejects weight=0.2 (below 0.3) and weight=3.0 (above 2.5) — verified with unit tests checking constraint definition
- NULLs: accuracy_score, evaluation_date, actual_outcome nullable (predictions may not yet be evaluated)
- rolling_accuracy, last_mutation_date on agent_weights are nullable (may not have enough history yet)
- agent_id indexed on scores and memory tables (FK-like usage)

## Expected runtime on t3.large
- Migration: < 1 second (3 CREATE TABLE statements + 1 idempotent index creation)
- Tests: < 5 seconds (DB introspection + model unit tests)
